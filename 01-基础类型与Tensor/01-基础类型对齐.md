# 01 · 基础类型对齐

> 范围:`ScalarType`、`TypeMeta`、`SymInt`、`Layout`、`Storage`、`DispatchKey{,Set}`、`pointer (intrusive_ptr)` 等 libtorch 基础类型在 Paddle compat 层的对齐工作。

## 1. 主题概述

PyTorch 的 C++ 表面建立在一组**位于 `c10/core/` 与 `c10/util/`** 的基础类型上 — dtype 投影(`ScalarType`)、dtype 运行时句柄(`TypeMeta`)、符号整数(`SymInt`)、内存布局(`Layout`、`MemoryFormat`)、底层存储(`Storage`)、分发系统(`DispatchKey{,Set}`)、引用计数智能指针(`intrusive_ptr`)。这些类型在 PyTorch 中互相耦合,**几乎所有 ATen 接口的签名都依赖至少一个**。

Paddle 内部有自己的等价物 — `phi::DataType`、`phi::Place`、`phi::backends::gpu::*`、`Layout` 等,但布局、命名、能力都与 PyTorch 不一致。compat 层的工作就是在不修改 Paddle 内核的前提下,**在 `paddle/phi/api/include/compat/` 中提供一组与 PyTorch 同名、同行为、同 ABI 的薄封装**,把请求转发到 Paddle 内部实现。

难点主要在三处:

1. **类型语义并不总是 1:1** — 例如 PyTorch 的 `SymInt` 是支持符号表达的包装类,而 Paddle 不需要符号执行能力,所以最初用 `int64_t` 别名,后来因隐式转换歧义提升为独立类(#78807)。
2. **dtype 的"实体化"差异** — PyTorch 用 `TypeMeta`(全局表项 + 运行时 ID)做 dtype,而 Paddle 用 `phi::DataType` 枚举。`TensorOptions::dtype()` 必须返回 `TypeMeta`,因此需要补全 `typeid.{h,cpp}` 整套机制(#78257)。
3. **`intrusive_ptr` 引用计数语义** — `weak_use_count()` 在 PyTorch 是 weak refs 实数,在 Paddle `shared_ptr<DenseTensor>` 体系下没有直接对应,只能做"最低限度的可观察对齐"(#78809)。

## 3. 关键 PR 深度分析

### 3.1 `ScalarType` 从单点 helper 到完整对齐 — #77303、#78581、#78806、#78662

#### 背景

PyTorch 的 `c10::ScalarType` 是一个 `enum class : int8_t`,枚举每个 dtype(Byte / Char / Short / Int / Long / Half / Float / Double / ComplexHalf / BFloat16 / Bool / QInt8 / QUInt8 / BFloat16 / Float8_e4m3fn / Float8_e5m2 / Float4_e2m1fn_x2 / Bits1x8 …)。围绕它有一族 **静态 helper 函数**:

```cpp
constexpr bool isFloatingType(ScalarType t);
constexpr bool isIntegralType(ScalarType t, bool includeBool);
constexpr bool isComplexType(ScalarType t);
constexpr bool isQIntType(ScalarType t);
constexpr bool isBitsType(ScalarType t);
constexpr bool isBarebonesUnsignedType(ScalarType t);
constexpr bool canCast(ScalarType from, ScalarType to);
constexpr ScalarType toQIntType(ScalarType t);
constexpr ScalarType toUnderlying(ScalarType t);
constexpr ScalarType toRealValueType(ScalarType t);
constexpr ScalarType toComplexType(ScalarType t);
```

#### 改动

- **#77303**(起步,+131 行)— 新增 3 个最常用 helper:`is_complex()`、`is_floating_point()`、`is_signed()`,直接作为 `TensorBase` 的成员方法。同时在 `c10/core/ScalarType.h` 中铺起 `ScalarType` 枚举体系。配套测试 `c10_ScalarType_test.cc`(+92 行)。
- **#78581**(占位枚举,+541/-99 行)— 引入完整 dtype 体系所需的占位类型头文件:`c10/util/Float4_e2m1fn_x2.h`、`Float8_e4m3fnuz.h`、`Float8_e5m2fnuz.h`、`Float8_e8m0fnu.h`、`bits.h`、`qint8/qint32/quint2x4/quint4x2/quint8.h`(每个 ~40 行的最小可编译包装),让 `ScalarType` 枚举的所有变体都有对应类型可被 `template<>` 匹配。
- **#78806**(拆自 #78707,+174 行)— 补齐前述全部 11 个 helper 函数,统一在 `c10/core/ScalarType.h` 与 `torch/headeronly/core/ScalarType.h` 中。其中 `canCast(from, to)` 实现 PyTorch 的安全转换规则:复数→实数不允许、浮点→整数不允许、非 Bool→Bool 不允许等。
- **#78662**(headeronly 化,+912/-561 行)— 把可纯 header 化的 `ScalarType` / `DeviceType` / `TensorAccessor` 拆到 `paddle/phi/api/include/compat/torch/headeronly/`,这样外部 header-only 工具库(如 DeepGEMM 的 C++ template kernel)可以**只 include headeronly 子目录**,不依赖 Paddle 主仓的 cpp 实现,大幅降低 include 传染面。

#### 效果

测试仓 [#19](https://github.com/PFCCLab/PaddleCppAPITest/pull/19)(`ScalarTypeTest.cpp` +106 行)+ [#46](https://github.com/PFCCLab/PaddleCppAPITest/pull/46) 中 `ScalarTypeFunctionsTest.cpp` 全部 `MATCH`(参见 #46 PR 描述中的 100% diff 输出)。这套 helper 是后续 `TensorOptions`、`canCast` 检查、`promote_types`、量化类型分发的基础。

### 3.2 引入 `TypeMeta` 替代裸 `ScalarType` — #78257

#### 背景

PyTorch 中,`TensorOptions::dtype()` 返回的是 `c10::TypeMeta`,不是 `ScalarType`:

```cpp
// PyTorch
auto opts = at::TensorOptions().dtype(at::kFloat);
TypeMeta meta = opts.dtype();  // 不是 ScalarType
```

`TypeMeta` 是一个 16 字节的运行时句柄,内部存着 `TypeMetaData*`,后者包含:类型大小、对齐、ctor/dtor/copy 函数指针、name 字符串、是否 fundamental 等元数据。这套机制是 PyTorch C++ 类型反射的基础。

早期 Paddle compat 层用 `ScalarType` 顶替 `TypeMeta`,导致下游代码 `auto m = opts.dtype(); m.itemsize();` 调用就编不过(`ScalarType` 没有 `.itemsize()`)。

#### 改动(+1570/-95 行,25 个文件)

1. **`c10/util/typeid.h`**(+646 行)+ **`typeid.cpp`**(+98 行) — 完整移植 `TypeMeta`、`TypeMetaData`、`TypeIdentifier` 模板机器,包括 `_PaddleFundamentalType`、`GuardLongUniqueDummy`、`Uninitialized` 等内部 helper。
2. **`c10/util/TypeIndex.h`**(+95 行) — 编译期类型 id 生成。
3. **`c10/core/ScalarTypeToTypeMeta.h`**(+54 行) — 在 `ScalarType` ↔ `TypeMeta` 之间双向映射,确保旧代码用 `ScalarType` 也兼容。
4. **`c10/core/TensorOptions.h`**(+53/-8) — `dtype()` 返回 `TypeMeta`、`dtype_opt()` 维持 `optional<ScalarType>` 兼容。
5. **`c10/core/DefaultDtype.h`**(+7/-4) — `get_default_dtype()` 返回 `TypeMeta`。
6. 所有 `ops/*.h` 中创建 tensor 的接口(`empty`、`full`、`zeros`、`ones`、`arange`、`eye`、`new_empty/full/ones/zeros`)统一改成接收 `TypeMeta`,内部转 `ScalarType` 再传给 Paddle。
7. **`utils/scalar_type_conversion.h`**(+6 行) — 增加 `TypeMeta` → `phi::DataType` 直接通路。

#### 效果

- 测试仓 [#56](https://github.com/PFCCLab/PaddleCppAPITest/pull/56) 贡献 `type_meta.md`(+92 行)文档,#46 中 PR 描述的 100% diff 表格中 `TensorOptionsTest` / `DefaultDtypeTest` 全部 MATCH。
- 后续 #78580 中把 `c10/util/typeid.{h,cpp}` 的部分实现进一步移到 cpp 文件以减少 inline 体量。
- `c10_TypeMeta_test.cc`(+282 行)在主仓覆盖运行时类型识别、`fromScalarType` / `toScalarType` 往返、`itemsize` 一致性等。

### 3.3 `SymInt` 从别名提升为类 — #77301、#78807

#### 背景与起步(#77301,+207/-36 行)

PyTorch 的 `c10::SymInt` 是支持符号表达的 `int64_t` 包装,用于 trace / shape-symbolic execution。它的出现让 `sym_size()` / `sym_stride()` / `sym_numel()` / `sym_sizes()` / `sym_strides()` 这些 `Tensor` 成员可以返回**未求值的**表达。

Paddle 不做 shape symbolic execution,所以 #77301 起步时把 `c10::SymInt = int64_t`,直接转发给 `IntArrayRef` 体系。这个简化能让 80% 的下游代码编过,但当下游同时使用 `SymIntArrayRef` 和 `IntArrayRef` 重载时,就会因为 "类型相同" 触发 ambiguous overload。

新增 5 个成员方法:`sym_size(dim)`、`sym_stride(dim)`、`sym_sizes()`、`sym_strides()`、`sym_numel()`,以及 `c10_SizesAndStrides_test.cc`(+187 行)单测。

#### 升级为独立类(#78807,+42/-13 行,9 个文件)

```cpp
// c10/core/SymInt.h
class SymInt {
 public:
  SymInt(int64_t v) : v_(v) {}
  operator int64_t() const { return v_; }
  int64_t guard_int() const { return v_; }
  std::optional<int64_t> maybe_as_int() const { return v_; }
 private:
  int64_t v_;
};

// c10/core/SymIntArrayRef.h
// 旧: using SymIntArrayRef = IntArrayRef;
// 新:
using SymIntArrayRef = ArrayRef<SymInt>;
```

关键适配:

- `TensorBase::sym_sizes()` / `sym_strides()` 用 `reinterpret_cast<const SymInt*>(IntArrayRef.data())` 构造 `SymIntArrayRef`,在 `SymInt` 物理布局为单个 `int64_t` 的前提下零拷贝。
- `ATen/TensorIndexing.h` 补 `int64_t` → `TensorIndex` 的隐式构造,消除原来依赖 `SymInt = int64_t` 的索引代码。
- `ops/split.h`、`ops/tensor_split.h`、`ops/unflatten.h` 修正 `_symint` 重载的内部 `SymIntArrayRef → IntArrayRef` 转换。
- `c10/util/ArrayRef.h::_PD_ToPaddleIntArray()` 用 `reinterpret_cast` 适配 `SymInt` 元素。

测试仓 [#18](https://github.com/PFCCLab/PaddleCppAPITest/pull/18) `TensorTest.cpp` +74 行,覆盖 `sym_size`/`sym_stride`/`sym_sizes`/`sym_strides`/`sym_numel` 的 5 个 case。

#### 效果

下游 `_symint` 后缀重载完全可编译;在测试仓 #46 的 100% diff 表格中 `EmptyStridedTest`(`sym_sizes` 主要使用者)MATCH。

### 3.5 `pointer / intrusive_ptr` — #77388、#78070

#### `pointer` 起步(#77388,+318 行,6 个文件)

PyTorch 中 `Tensor` 通过 `c10::intrusive_ptr<TensorImpl>` 持有底层实现。`intrusive_ptr` 的关键能力:`use_count()`(强引用计数)、`weak_use_count()`(弱引用计数)、`is_same(other)`(对象同一性)。

`#77388` 在 `TensorBase` 上补出以下方法 — 但**实现暂时绕过 `TensorImpl`**:

```cpp
bool TensorBase::is_same(const TensorBase& other) const;
int64_t TensorBase::use_count() const;
int64_t TensorBase::weak_use_count() const;  // 暂时固定返回 0
bool TensorBase::is_contiguous_or_false() const;
std::string TensorBase::toString() const;
void TensorBase::print() const;
```

PR 描述:"PyTorch 的 impl 指针由 `c10/util/intrusive_ptr.h` 实现,在 `c10/core/TensorImpl.h` 中初始化,这里先实现依赖大多已经实现的 `intrusive_ptr.h`(基本上是复制粘贴的,只处理了 pre-commit)。"

`TensorImpl.h` 与 `DispatchKey.h` 都未跟进,留待 #77416 / #78070。

#### `intrusive_ptr` 完整移植(#78070,+442 行)

`#78070` 在加 `Generator` 时**顺带**补齐 `c10/util/intrusive_ptr.h`(+442 行),提供完整的 `intrusive_ptr_target` 基类、强弱引用计数、move/copy 语义。这样后续 `Generator`、`Storage` 等内部对象都可以正确遵循 PyTorch 的引用语义。

#### `weak_use_count` 对齐(#78809,+8/-11 行)

PR 描述里写得很直接:"`weak_use_count` 是 PyTorch `intrusive_ptr` 的特性,无法完全对齐,这里只对齐部分功能"。

具体做法:

```cpp
// 旧
int64_t TensorBase::weak_use_count() const { return 0; }
// 新:对齐 PyTorch 的可观察行为:live TensorImpl 至少 weak refs = 1
int64_t TensorBase::weak_use_count() const {
  return defined() ? 1 : 0;
}
```

代价小、效果有限,但消除了测试仓的一组 mismatch。该 PR 拆自 #78707,作者明确表示"不确定是否合入",但导师审过认为可以先把可观察语义对齐。

### 3.6 `DispatchKey{,Set}` — #77399(OPEN)、#77416(CLOSED)、#78070

PyTorch 用 `c10::DispatchKey`(64 个枚举值,覆盖 backend、autograd、quantized、sparse、functorch、vmap 等正交维度)+ `DispatchKeySet`(64 位掩码) 实现 op 分发。`TensorImpl` 上的 `key_set_` 字段控制路由。

- **#77416**(CLOSED,+854 行) — 起步尝试,在 `TensorBase` 上加 `is_sparse()` / `is_sparse_csr()` / `is_quantized()` / `is_meta()` / `is_inference()` / `is_nested()`,以及对应的 `ops/sparse_coo_tensor.h` / `sparse_csr_tensor.h`。导师建议拆分,**因此被关闭**,稀疏部分单独发为 #77581,DispatchKeySet 部分留给 #77399。
- **#77399**(OPEN,+57 行) — 仅在 `TensorBase` 上补 `_is_zerotensor`、`_set_zero`、`is_conj`、`_set_conj`、`is_neg`、`_set_neg` 接口。但 PR 描述提出疑问:"`key_set_` 属性在 `c10/core/TensorImpl.h` 中定义,是否有必要在 `TensorBase.h` 下新增该属性?" — 设计未定型,**停留为 OPEN**。
- **#78070** — 顺带补齐了完整的 `c10/core/DispatchKey.h`(+598 行,所有 64 个枚举值)与 `DispatchKeySet.h`(+735 行,完整 64 位掩码运算),并提供 `c10_DispatchKeySet_test.cc`(+439 行)+ `c10_DispatchKey_test.cc`(+248 行)单测。

虽然 #77399 没合入,**最关键的类型(`DispatchKey` + `DispatchKeySet`)已经通过 #78070 落地**,Tensor 侧的 `key_set` 字段就成了 nice-to-have。
