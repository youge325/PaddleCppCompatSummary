# 02 · CI 与编译

> 范围:Linux ABI 兼容性检查、flashmla 编译修复、paddlecodec 编译适配、deepep legacy API 清理、FA 编译线程数调优、Python API 文档修复。

## 1. 主题概述

compat 层做的是**长期承诺**:`c10::` / `at::` / `torch::` / `caffe2::` 命名空间下导出的接口,后续 PR 不应误删,否则下游已编译好的二进制不再可用。

但 Paddle 项目体量巨大,几百名 contributor 并发提交,很容易有人**在不知情的情况下**删除一个 compat 符号(比如以为某个 `at::cuda::xxx` 是无用代码)。

解决方案是 CI 自动化:**比对 base wheel 与 PR wheel 的动态符号导出表**,只要 PR 删了任何 protected ABI 符号,直接 fail,要求导师手动 approve。这是 #78831 的工作。

此外,本章也覆盖**下游编译触发的零散修复**:flashmla / paddlecodec / DeepEP 等真实下游报错后产出的 hotfix PR。

## 3. 关键 PR 深度分析

### 3.1 ★ Linux ABI 兼容性检查 — #78831(+833/-1 行,4 个文件)

这是**项目工程治理的核心 PR** — 把"compat 层稳定承诺"变成 CI 自动化拦截。

#### 设计原则

PR 描述非常清晰:

> 该检查会比较 base wheel 与 PR wheel 中 Paddle runtime libraries 的动态符号,**只在 PR 删除 compat-facing ABI 符号时失败**,新增符号和非 compat 层符号不会触发失败。

也就是说:

- **新增符号**(下游需要的新接口) → 不拦截
- **修改非 compat 命名空间符号**(`paddle::`、`phi::`) → 不拦截
- **删除 `c10::`、`at::`、`torch::`、`caffe2::` 的符号** → 拦截,要求导师 approve

#### 检查范围

`tools/check_abi_compatibility.py`(+520 行)的核心逻辑:

```python
# 提取动态符号
def extract_dyn_syms(libpath):
    out = subprocess.check_output(['readelf', '--dyn-syms', '-W', libpath])
    return parse_readelf_output(out)

PROTECTED_NAMESPACES = ('c10::', 'at::', 'torch::', 'caffe2::')
PROTECTED_LIBS = ['libpaddle.so', 'libphi.so', 'libphi_core.so', 'libphi_gpu.so']

def check(base_wheel_dir, pr_wheel_dir):
    deleted = []
    for lib in PROTECTED_LIBS:
        base_syms = extract_dyn_syms(f'{base_wheel_dir}/{lib}')
        pr_syms   = extract_dyn_syms(f'{pr_wheel_dir}/{lib}')
        for sym in base_syms - pr_syms:
            if any(sym.startswith(ns) for ns in PROTECTED_NAMESPACES):
                deleted.append((lib, sym))
    return deleted
```

被忽略的情形:

- 弱符号(`weak` 链接,可被覆盖)
- 未定义符号(`UND`,只是引用,本来就不在该库内)
- local 符号(不导出)
- 第三方符号(`__cxx_*`、`std::*` 等)
- 非 protected 命名空间

#### 单测覆盖(+295 行)

`test/tools/test_check_abi_compatibility.py` 13 个 case:

- 符号纯新增 → pass
- 添加 + 删除 compat 符号 → fail,列出删除项
- 删除非 compat 符号 → pass
- 弱符号删除 → 忽略
- 未定义符号删除 → 忽略
- local 符号变化 → 忽略
- 缺失 PR library → fail("PR 删除了一个库")
- 第三方符号删除 → 忽略
- 等等

#### CI 集成

`ci/static_check.sh`(+17 行)添加调用:

```bash
# 比对 build/dev_whl/*.whl 与 build/pr_whl/*.whl
python tools/check_abi_compatibility.py \
    --base-wheel build/dev_whl \
    --pr-wheel  build/pr_whl
if [ $? -ne 0 ]; then
  echo "ABI symbols removed; requires SigureMo or BingooYang approval"
  exit 1
fi
```

`ci/check_approval.sh`(+1/-1) 把 ABI checker 加入"需 approval" 规则。

#### 重要细节

PR 描述里特别说明了**两个工程权衡**:

1. **"ABI 符号变动" 统一视为"删除"** — 如果同名符号但 mangled name 变了(比如参数类型变了),严格说是"修改"而非"删除",但 checker 把它当"删除"处理。这是 conservative 选择,可能略有 false-positive 但绝不会漏掉真删除。

2. **触发 approval 的条件**严格:**只有 checker 实际检测到 protected ABI 删除时**,Static Check 才要求 [@SigureMo](https://github.com/SigureMo) 或 [@BingooYang](https://github.com/BingooYang) approve。修改 checker / CI 文件本身**不会**触发 approval — 否则改 CI 都要导师签字,工作流会很难推进。

#### 影响

这个 PR 在 2026-05-01 合入后,从此 compat 层的任何符号删除都需要审批。Paddle 主干上后续看到的"compat ABI removed; needs approval"消息就来自这里。

### 3.2 `flashmla` 编译修复 — #78550(+142/-92 行,11 个文件)

#### 背景

FlashMLA 是 DeepSeek 提出的高性能 MLA(Multi-head Latent Attention)kernel,被 PaddleNLP 用作 Paddle 上的 MLA 实现。它使用 `at::cuda::is_available()` 在编译期判断 CUDA 可用性。

compat 层早期把 `is_available` **同时**通过两个路径暴露:

```cpp
// torch/cuda.h(旧)
namespace at::cuda {
  using torch::cuda::is_available;  // 重复导出到 at::cuda
}

// ATen/cuda/CUDAContext.h
namespace at::cuda {
  bool is_available();  // 独立声明
}
```

`#include <ATen/cuda/CUDAContext.h>` 之后,两个声明撞车,FlashMLA 编译报 redefinition。

#### 改动(拆自 #78484)

**1. `torch/csrc/api/include/torch/cuda.h`** 删除重复导出:

```cpp
namespace at::cuda {
-  using torch::cuda::device_count;
-  using torch::cuda::is_available;
  using torch::cuda::synchronize;
}
```

修复策略:`is_available` 在 `ATen/cuda/CUDAContext.h` 中独家声明,不再 forward 到 `at::cuda`。下游统一用 `torch::cuda::is_available()`(更通用的入口)。

**2. 更新测试**:

- `ATen_TensorAccessor_test.cc`(+1/-1)
- `compat_basic_test.cc`(+1/-1)
- `c10_storage_test.cc`(+5/-4)

把硬编码的 `at::cuda::is_available()` 改为 `torch::cuda::is_available()`。

**3. Stream 比较运算符行为对齐** — `c10/core/Stream.h`(+2/-3)、`Stream.cpp`(+5/-3):

- `Stream::operator==` 现按 (device, stream_id) 严格比较
- `native_handle()` 在 CPU 设备上抛 `STD_TORCH_CHECK(false, ...)` 而非 segfault

**4. 精简 `ScalarType` 实现** — `c10/core/ScalarType.h`(+35/-13):

移除部分尚未完全对齐的 `ScalarType` 代码(将在 #78581、#78806 中重新引入,本 PR 不带这个负担)。

**5. 测试** — `c10_ScalarType_test.cc`(+47/-62)同步调整断言。

#### 效果

FlashMLA 编译通过。

### 3.3 `deepep` legacy API 清理 — #78549(+19/-165 行,9 个文件)

#### 背景

早期为兼容 DeepEP 老版本接口,compat 层 添加了几个临时 forwarder(在 `record_stream.h`、`Event.h`、`Stream.h`、`CUDAStream.h` 中):

```cpp
// 旧:为 DeepEP 0.x 提供的临时接口
namespace at {
  inline void cuda_record_stream(/* legacy signature */);
}
```

DeepEP 自己已经迁移到上游 PyTorch 风格的新接口,这些 legacy forwarder 没人用了。

#### 改动(拆自 #78484)

净删除 165 行:

- `ATen/core/TensorBody.h`(-7 行) — 删 legacy `cuda_record_stream` Tensor 方法
- `ATen/ops/record_stream.h`(-27 行) — 删 legacy 重载
- `c10/core/Event.h`(-19 行) — 删 legacy `Event(int)` 构造
- `c10/core/Stream.h`(+4) — 把 `Stream` 暴露到 `at::Stream`(`namespace at` 下接口自动暴露到 `namespace torch`,DeepEP 用 `torch::Stream` 即可)
- `c10/cuda/CUDAStream.h`(-28 行) — 删 legacy `getCurrentCUDAStreamFromPaddle` 等命名
- `c10/cuda/CUDAStream.cpp`(+15) — 新加 `at::Stream` 转 `c10::Stream` 的转发实现
- 测试:`ATen_record_stream_test.cc`(-18)、`c10_Event_test.cc`(-31)、`c10_Stream_test.cc`(-35) — 删 legacy case

#### 效果

ABI 表面变小,头文件更清爽。这是 ABI checker 上线前的最后一次清理。

### 3.7 PaddleCppAPITest 仓库的 CMake 修复 — PFCCLab#8

#### 背景

测试仓在 Ubuntu 24.04 + cmake 3.28.2 + gcc 13.3.0 环境下 `ctest` 报"Could not find executable torch_TensorTest"。

PR 描述附带详细日志:cmake 在新版下,`add_test` 默认按构建配置子目录(`Release/` / `Debug/`)查找可执行文件,但测试仓的构建产物在根目录。

#### 改动

`cmake/build.cmake`(+1/-1):

```diff
-add_test(NAME ${name} COMMAND ${name})
+add_test(NAME ${name} COMMAND $<TARGET_FILE:${name}>)
```

显式指定 `$<TARGET_FILE:...>` 让 cmake 在构建产物的实际路径中找。

PR 描述附带改动前后的 `ctest` 完整输出,显示从全部 "Not Run" 到全部 PASS 的转变。
