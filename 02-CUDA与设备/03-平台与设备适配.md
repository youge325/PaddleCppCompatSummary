# 03 · 平台与设备适配

> 范围:仅 CPU 编译路径的条件编译适配、Windows compat 测试构建支持、DCU(海光)HIP native API 适配、XPU(昆仑芯)device 索引解析。

## 1. 主题概述

compat 层最初聚焦 **Linux + NVIDIA CUDA**,因为绝大多数下游(DeepEP / DeepGEMM / FlashMLA / FastDeploy)首要部署目标都是这个组合。但随着工作推进,在三个方向上出现了适配需求:

| 平台 / 设备 | 触发原因 | 状态 |
|---|---|---|
| **Windows** | 内部用户编译 compat 测试时报符号导出错误 | #78670 合入(但默认不强制) |
| **DCU (HIP)** | 海光 DCU 设备需要 HIP runtime 替代 CUDA | #78595 合入 |
| **XPU** | 昆仑芯 XPU 设备 index 解析 | #78647 合入 |

跨平台适配的核心策略是:**接口签名不变,内部用 `#ifdef` 切换 backend**。compat 层不为每个平台维护独立头/源文件树。

## 3. 关键 PR 深度分析

### 3.1 DCU (HIP) native API 适配 — #78595(+587/-80 行,28 个文件)

#### 背景

海光 DCU 是国产 GPU,软件栈用 **HIP**(AMD 的 ROCm 衍生,与 CUDA API 接近但函数名/类型名不同 — `hipStream_t` vs `cudaStream_t`、`hipDeviceSynchronize` vs `cudaDeviceSynchronize`)。Paddle 通过 `PADDLE_WITH_HIP` 宏在编译时切换 backend。

compat 层之前所有 CUDA 路径直接用 `cudaStream_t` / `cudaEvent_t` / `cudaDeviceSynchronize` 等 CUDA 原生类型/函数,在 DCU 编译下失败。

#### 改动策略

采取**保留接口签名、内部用 typedef + `#ifdef` 切换**的策略:

```cpp
// c10/cuda/CUDAStream.h
#if defined(PADDLE_WITH_HIP)
using gpuStream_t = hipStream_t;
using gpuEvent_t = hipEvent_t;
using gpuError_t = hipError_t;
#define gpuStreamCreate hipStreamCreate
#define gpuStreamDestroy hipStreamDestroy
#define gpuDeviceSynchronize hipDeviceSynchronize
// ...
#else
using gpuStream_t = cudaStream_t;
using gpuEvent_t = cudaEvent_t;
using gpuError_t = cudaError_t;
#define gpuStreamCreate cudaStreamCreate
// ...
#endif
```

然后 compat 层的 `c10::cuda::CUDAStream::stream()` 返回 `gpuStream_t`,在 CUDA 下展开为 `cudaStream_t`,DCU 下展开为 `hipStream_t` — **接口签名不变**,源代码可以同一份覆盖两种 backend。

#### 涉及文件

28 个文件 +587/-80:

- **`ATen/core/TensorBody.h`**(+9/-3) — `tensor.record_stream(Stream)` 适配 HIP
- **`ATen/cuda/CUDAContext.h`**(+2/-1)、**`CUDAContextLight.h`**(+10/-6)、**`CUDADataType.h`**(+31/-3) — handle 类型切换
- **`ATen/cuda/CUDAEvent.h`**(+52) — `cudaEvent_t` → `gpuEvent_t`
- **`ATen/ops/record_stream.h`**(+13) — HIP 路径
- **`c10/core/Event.h`**(+14/-2)、**`Stream.cpp`**(+17/-18) — Event/Stream HIP 适配
- **`c10/cuda/CUDAException.h`**(+13/-1) — 错误类型双轨
- **`c10/cuda/CUDAStream.{h,cpp}`**(+40 + +40) — Stream wrapper HIP 适配
- **`phi/backends/gpu/rocm/rocm_info.cc`**(+6/-1) — 在 Paddle ROCM backend 补 `GetDeviceProperties` 等接口
- **`test/CMakeLists.txt`**(+9) — DCU 测试编译目标

#### 测试

PR 描述提到"等待 #78580 合入后再将单测纳入编译"。13 个测试文件 +200 余行修改 — 大部分是把测试中硬编码的 `cudaStream_t` 改为 `gpuStream_t`,把 `cudaSetDevice` 改为 `phi::backends::gpu::SetDeviceId`(后者内部已 dispatch CUDA/HIP)。

### 3.2 XPU device 索引解析 — #78647(+150/-25 行,7 个文件)

#### 背景

昆仑芯 XPU 是国产 AI 加速器,Paddle 通过 `PADDLE_WITH_XPU` 支持。PyTorch 的 `c10::Device("xpu")` 在不显式指定 `index` 时,**应解析到当前 XPU device**(类似 `c10::Device("cuda")` → 当前 CUDA device)。

compat 层早期实现把没指定 index 的 XPU device 解析为 `XPU:-1`(无效),导致下游创建 Tensor 时 `tensor.device().is_xpu()` 返回 true,但 `tensor.device().index()` 是 -1,后续操作崩溃。

#### 改动

- **`c10/core/Device.h`**(+2/-2) — `Device::Device(DeviceType type)` 构造函数中,XPU 路径走 `phi::backends::xpu::GetCurrentXpuId()` 解析当前 device index。
- **`ATen/core/TensorBody.h`**(+11/-14)、**`ATen/ops/to.h`**(+4/-4) — Tensor → XPU 设备转换路径正确传 device index。
- **测试**:
  - `ATen_basic_test.cc`(+58) — XPU device 基础查询、Tensor 创建
  - `ATen_empty_test.cc`(+21/-2) — `at::empty({3}, at::TensorOptions().device("xpu"))` 路径
  - `ATen_to_test.cc`(+37/-3) — `tensor.to("xpu")` 路径
  - `c10_Device_test.cc`(+17) — Device 解析

### 3.3 Windows compat 测试构建 — #78670(+324/-222 行,24 个文件)

#### 背景

Paddle 在 Windows 上发布 wheel,理论上 compat 层应同样可在 Windows 编译。但内部用户尝试在 Windows 构建 compat 单测时报大量**符号导出错误**(MSVC 的 `__declspec(dllexport)` 与 `__declspec(dllimport)` 严格要求,而 Linux 默认全部 export)。

PR 描述非常坦诚:"在 Windows 平台编译兼容层测试,全都是符号导出问题,**不确定要不要合入,暂时没有这个需求**"。最终在导师 review 后还是合入,但**默认 CMake 配置不强制开启 Windows 测试**。

#### 主要改动

**1. 符号导出注解** — 几乎每个 compat 头文件都需要在公开符号上加 `PADDLE_API`:

```cpp
// c10/util/typeid.h(+13/-7)
class PADDLE_API TypeMeta { ... };

// c10/core/Device.h(+3/-2)
class PADDLE_API Device { ... };

// c10/cuda/CUDAFunctions.h(+2/-2)
PADDLE_API DeviceIndex device_count() noexcept;
PADDLE_API bool is_available() noexcept;
```

**2. `c10/core/DefaultDtype.cpp`**(+50 行)新建

把原本在 `DefaultDtype.h` 中 inline 的 thread-local 静态变量定义搬到 cpp,避免 MSVC 在多个 TU 实例化 thread-local 时报重复定义。

**3. 库加载路径**

- `paddle/phi/backends/dynload/dynamic_loader.cc`(+6/-3) — Windows 上 dlopen 用 `LoadLibraryW`、PATH 解析差异。
- `paddle/phi/core/platform/cuda_graph_with_memory_pool.{h,cc}`(+3/-12 + +17) — CUDA graph 在 Windows 下的额外初始化分支。

**4. ERROR 宏污染规避**

Windows 头文件 `wingdi.h` 定义 `#define ERROR 0`,与 compat 层内部枚举 `enum class State { START, ERROR, ... }` 冲突。解决方案:**改名内部枚举**,所有内部状态枚举改为 `kStart`、`kError`、`kIndexStart` 等小写 + `k` 前缀的命名,**保持对外行为完全一致**。具体涉及 `c10/core/Device.h`(+3/-2)、`Device.cpp` 等。

**5. CMake 拆分**

`test/cpp/compat/CMakeLists.txt`(+81/-84)— 把测试目标按 Windows 可编译性分组,Windows 上跳过的目标加 `if(NOT WIN32)` 包裹(主要是涉及 `fork()` / Unix socket 的测试)。

**6. `paddle/phi/api/include/compat/CMakeLists.txt`**(+1) — 把 `DefaultDtype.cpp` 加入编译列表。

#### 涉及测试

`test/cpp/compat/ATen_factory_default_dtype_test.cc`(+4/-3)、`c10_TensorOptions_test.cc`(+29/-3)、`c10_generator_impl_test.cc`(+17/-17)、`torch_library_test.cc`(+2/-4) — 主要是把硬编码的 `thread_local` 用法调整,确保 MSVC 编译过。

### 3.4 仅 CPU 编译适配 — #78580

**背景**:compat 层最初只考虑 `PADDLE_WITH_CUDA` 或 `PADDLE_WITH_HIP` 两条路径,头文件中存在大量**隐式 else 分支**(即 `#ifdef PADDLE_WITH_HIP ... #else ...`,else 分支默认当作 CUDA 处理)。在**纯 CPU 编译**(`!CUDA && !HIP`)路径下,这些 else 分支会把 CUDA 原生类型/函数暴露出来,导致编译失败。

**改动**:#78580 中通过**显式条件编译**堵上这些缺口:

**1. `CUDAContextLight.h`** — 把隐式 else 改为显式 `#elif`:

```cpp
// 修改前:#else 默认进入 CUDA 分支
-#else
+#elif defined(PADDLE_WITH_CUDA)
 using CUDAContextDeviceProp = cudaDeviceProp;
```

并把一批 CUDA-only API 声明用 `#if defined(PADDLE_WITH_CUDA) || defined(PADDLE_WITH_HIP)` 包裹,防止 CPU-only 编译时引用不存在的类型:

```cpp
+#if defined(PADDLE_WITH_CUDA) || defined(PADDLE_WITH_HIP)
 CUDAContextDeviceProp* getCurrentDeviceProperties();
 int warp_size();
 // ... 其他 CUDA-only API
 CUDAContextSolverHandle getCurrentCUDASolverDnHandle();
```

**2. `PhiloxUtils.cuh`** — CUDA `__host__ __device__` 属性在 CPU-only 路径下不存在,加条件编译切换:

```cpp
+#if defined(PADDLE_WITH_CUDA) || defined(PADDLE_WITH_HIP)
 __host__ __device__ __forceinline__ std::tuple<uint64_t, uint64_t> unpack(
     at::PhiloxCudaState arg) {
+#else
+inline std::tuple<uint64_t, uint64_t> unpack(at::PhiloxCudaState arg) {
+#endif
```

**3. 测试侧同步适配**

`ATen_CUDAContext_test.cc` 中把 `#include <c10/cuda/CUDAStream.h>` 和 GPU-only 测试用 `#if defined(PADDLE_WITH_CUDA) || defined(PADDLE_WITH_HIP)` 包裹,CPU-only 构建下这些测试直接跳过或验证 throw 行为:

```cpp
TEST(CUDAFunctionsTest, DeviceSynchronize) {
+#if defined(PADDLE_WITH_CUDA) || defined(PADDLE_WITH_HIP)
   ASSERT_NO_THROW(c10::cuda::device_synchronize());
+#else
+  ASSERT_THROW(c10::cuda::device_synchronize(), std::exception);
+#endif
}
```

同时删除 `cuda_test_utils.h`(原先用于运行时跳过无 CUDA 的测试,现在用编译期条件编译替代)。

**结果**:compat 层在纯 CPU 编译路径下**不再暴露 CUDA 原生符号**,测试也能在 CPU-only 构建中编译通过。这是后续 #78632 中提到的"CPU-only 编译失败"问题的**根因修复**。"

## 4. 与 PaddleCppAPITest 的对应关系

测试仓本身**不涉及** DCU / XPU / Windows 适配 — 它的 CI 只跑 Linux + CUDA(`.github/workflows/compile-test.yml` 在 #59 中引入)。本章节的工作都在主仓侧。

不过测试仓在 #60 中接入 `PADDLE_WITH_CUSTOM_KERNEL` 宏(在 [`04-PaddleCppAPITest仓库/`](../04-PaddleCppAPITest仓库/README.md) 详述),与本章节的"接口签名稳定、内部 backend 切换"思路一致 — 不同 backend 编译同一份测试代码。
